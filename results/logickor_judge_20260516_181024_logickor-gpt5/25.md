## C++
- **성격**: 시스템/고성능 애플리케이션용 범용 언어
- **장점**: 매우 높은 성능, 하드웨어 제어 용이, 방대한 생태계(게임/엔진/금융/임베디드)
- **특징**: 수동 메모리 관리 가능(스마트 포인터로 완화), 템플릿/제네릭 강력, 복잡도가 높음
- **주요 용도**: 게임 엔진, 고성능 서버, 데스크톱 앱, 임베디드, HFT 등

## JavaScript
- **성격**: 웹의 표준 스크립트 언어(브라우저 + 서버(Node.js))
- **장점**: 웹 프론트엔드 필수, 개발 속도 빠름, 생태계(NPM) 매우 큼
- **특징**: 동적 타입, 이벤트 루프 기반 비동기(프라미스/async-await), 런타임 성능은 엔진(V8 등)에 의존
- **주요 용도**: 웹 프론트, Node.js 백엔드, 서버리스, 크로스플랫폼(Electron 등)

## Go (Golang)
- **성격**: 서버/클라우드 인프라에 초점을 둔 실용적 언어
- **장점**: 단순한 문법, 컴파일/배포 쉬움(단일 바이너리), 동시성(고루틴/채널) 강력
- **특징**: 가비지 컬렉션(GC), 인터페이스 중심 설계, “명확하고 단순함”을 우선(표현력은 상대적으로 보수적)
- **주요 용도**: 마이크로서비스, 네트워크 서버, DevOps 도구, 클라우드 네이티브(K8s 생태계)

## Rust
- **성격**: 메모리 안전성과 성능을 동시에 추구하는 시스템 언어
- **장점**: GC 없이도 메모리 안전(소유권/빌림 검사), C/C++급 성능, 동시성 안전성 높음
- **특징**: 학습 난이도 높음(소유권 모델), 컴파일러가 많은 오류를 사전에 잡아줌, 패키지 매니저 Cargo 강력
- **주요 용도**: 시스템/임베디드, 고성능 서비스, 보안 민감 영역, WASM, C/C++ 대체

## Swift
- **성격**: Apple 플랫폼 중심의 현대적 언어(iOS/macOS 등)
- **장점**: 안전한 문법(옵셔널), 생산성 높음, Apple 생태계 최적화
- **특징**: ARC(참조 카운팅) 기반 메모리 관리, 함수형 요소/제네릭 지원, 앱 개발 경험 좋음
- **주요 용도**: iOS/iPadOS/macOS 앱, 일부 서버 사이드(SwiftNIO 등) 가능

## 🔵 C++

**"성능의 제왕"**

```cpp
#include <iostream>
int main() {
    std::cout << "Hello, World!" << std::endl;
    return 0;
}
```

**✅ 장점**
- 최고 수준의 실행 성능
- 하드웨어 직접 제어 가능
- 방대한 레거시 코드베이스

**❌ 단점**
- 복잡한 문법, 가파른 학습 곡선
- 메모리 오류 발생 가능성

**🎯 사용처:** 게임 엔진, OS, 임베디드, 고성능 컴퓨팅

---

## 🟡 JavaScript

**"웹의 언어"**

```javascript
const greet = (name) => `Hello, ${name}!`;
console.log(greet("World"));
```

**✅ 장점**
- 브라우저에서 바로 실행
- 프론트엔드 + 백엔드(Node.js) 모두 가능
- 거대한 생태계(npm)

**❌ 단점**
- 동적 타입으로 인한 런타임 오류
- `this` 바인딩 등 혼란스러운 부분

**🎯 사용처:** 웹 개발, 서버(Node.js), 모바일(React Native)

---

## 🔷 Go (Golang)

**"단순함의 미학"**

```go
package main
import "fmt"

func main() {
    fmt.Println("Hello, World!")
}
```

**✅ 장점**
- 배우기 쉬운 간결한 문법
- 강력한 동시성 지원 (goroutine)
- 빠른 컴파일 속도

**❌ 단점**
- 제네릭 지원 제한적 (최근 개선됨)
- 에러 처리가 다소 번거로움

**🎯 사용처:** 클라우드 인프라, 마이크로서비스, DevOps 도구

---

## 🟠 Rust

**"안전한 시스템 프로그래밍"**

```rust
fn main() {
    let name = "World";
    println!("Hello, {}!", name);
}
```

**✅ 장점**
- 컴파일 타임 메모리 안전성 보장
- C++ 수준의 성능
- 현대적인 패키지 관리(Cargo)

**❌ 단점**
- 가파른 학습 곡선 (소유권 개념)
- 컴파일 시간이 긴 편

**🎯 사용처:** 시스템 프로그래밍, WebAssembly, 블록체인

---

## 🟣 Swift

**"Apple 생태계의 핵심"**

```swift
let name = "World"
print("Hello, \(name)!")
```

**✅ 장점**
- 읽기 쉬운 현대적 문법
- 강력한 타입 안전성
- Xcode와의 완벽한 통합

**❌ 단점**
- Apple 플랫폼 중심
- ABI 안정성 이슈 (과거)

**🎯 사용처:** iOS/macOS 앱, 서버 사이드(Vapor)

## 1) C++가 좋은 상황 (최고 성능 + 하드웨어/저수준 제어)
### 예시
- **게임 엔진/그래픽스**: 렌더링 파이프라인, 물리 엔진, 실시간 처리(프레임 유지가 중요)
- **초저지연(ultra-low latency) 시스템**: 금융 트레이딩, 실시간 스트리밍 처리
- **임베디드/드라이버/로보틱스**: 하드웨어와 가깝게 붙고 메모리/성능 최적화가 중요한 경우
### 왜 C++?
- 런타임 오버헤드가 낮고, 세밀한 최적화와 시스템 자원 제어가 가능.

---

## 2) JavaScript가 좋은 상황 (웹 UI + 빠른 프로토타이핑 + 풀스택 가능)
### 예시
- **웹 프론트엔드**: 인터랙티브 UI(React/Vue/Svelte), SPA, 대시보드
- **가벼운 백엔드/서버리스**: API 서버(Node.js), Lambda/Cloud Functions
- **프로토타입을 빨리**: 기능 검증(MVP)이나 데모를 빠르게 만들어야 할 때
### 왜 JavaScript?
- 브라우저의 표준 언어라 프론트엔드는 사실상 필수. 생태계가 커서 “속도”가 나옴.

---

## 3) Go가 좋은 상황 (클라우드/서버/네트워크 + 단순함 + 동시성)
### 예시
- **마이크로서비스/API 서버**: 배포가 잦고 운영이 중요한 B2B 서비스
- **DevOps/플랫폼 도구**: 배포 도구, 모니터링 에이전트, CLI 툴(단일 바이너리 배포가 편함)
- **동시성 네트워크 서버**: 많은 연결을 처리하는 프록시, 게이트웨이
### 왜 Go?
- 빌드/배포가 단순하고(정적 링크 단일 바이너리), 고루틴으로 동시성 구현이 쉬움. 운영 친화적.

---

## 4) Rust가 좋은 상황 (성능은 C급, 안전성은 더 높게)
### 예시
- **메모리 안전이 최우선인 시스템**: 보안 취약점(Use-after-free 등) 위험을 줄여야 하는 컴포넌트
- **고성능 서버의 핵심 모듈**: 병목 구간만 Rust로 작성하거나, 전체를 Rust로 구성
- **WASM 모듈**: 브라우저/엣지에서 빠르고 안전한 모듈 실행
- **C/C++ 대체**: 기존 C/C++ 코드의 안정성을 끌어올리고 싶을 때(점진적 도입도 가능)
### 왜 Rust?
- GC 없이도 메모리 안전성을 컴파일 타임에 강제해, “성능 + 안정성”을 동시에 노리기 좋음.

---

## 5) Swift가 좋은 상황 (Apple 앱 개발 최적)
### 예시
- **iOS/iPadOS/macOS 앱**: 앱스토어 배포, 네이티브 UI/성능, Apple 프레임워크 활용
- **Apple 생태계 기능 연동**: HealthKit, ARKit, CoreML, SwiftUI 등
- **기존 Objective-C 프로젝트 현대화**: 점진적으로 Swift로 전환
### 왜 Swift?
- Apple 플랫폼에서 도구체인/프레임워크/런타임 지원이 가장 좋고, 생산성과 안전성 균형이 좋음(옵셔널, ARC).

```cpp
    }
    
private:

    std::vector<GameObject*> objects;
};
```

```rust
// 메모리 안전성이 컴파일 타임에 보장됨
fn transfer_funds(from: &mut Account, to: &mut Account, amount: u64) -> Result<(), Error> {
    // 컴파일러가 동시 접근 방지
    if from.balance < amount {
        return Err(Error::InsufficientFunds);
    }
    
    from.balance -= amount;
    to.balance += amount;
    
    Ok(())
}

// C++에서 흔한 실수 - Rust에서는 컴파일 에러
// let ptr = &mut data;
// let ptr2 = &mut data;  // ❌ 동시에 두 개의 mutable 참조 불가
```
