# XSS(교차 사이트 스크립팅) 방지 방법

## 1) 출력 인코딩(escaping) — 가장 중요
사용자 입력을 “그대로” HTML에 넣지 말고, **출력되는 위치(컨텍스트)에 맞게 인코딩**합니다.

- **HTML 본문 컨텍스트**: `<`, `>`, `&`, `"`, `'` 등을 엔티티로 이스케이프
- **HTML 속성 컨텍스트**: 따옴표/공백/특수문자 처리 포함(가급적 속성 값은 항상 따옴표로 감싸기)
- **URL 컨텍스트**: URL 인코딩 및 허용 스킴(예: `https:`)만 허용
- **JavaScript 컨텍스트**: 문자열/템플릿/JSON 등 문맥에 맞는 안전한 직렬화 사용(가능하면 “JS에 직접 삽입” 자체를 피하기)

> 권장: “입력 검증”보다 “출력 인코딩”이 XSS 방어의 중심입니다.

---

## 2) 안전한 DOM API 사용 (클라이언트 측)
브라우저에서 DOM을 조작할 때 사용자 입력을 HTML로 해석시키지 않는 것이 핵심입니다.

- 사용자 입력을 화면에 표시할 때는 **`textContent` / `innerText`** 사용
- 아래와 같은 **HTML 해석 API 사용 최소화/감사**  
  - `innerHTML`, `outerHTML`, `insertAdjacentHTML`
- URL 파라미터/해시를 DOM에 넣는 기능(라우팅, 검색어 표시 등)은 특히 주의

---

## 3) 신뢰할 수 있는 템플릿/프레임워크의 자동 이스케이프 활용
- React, Vue, Angular, 서버 템플릿(예: Django/Jinja2, Rails ERB 등)은 기본적으로 “출력 이스케이프”를 제공합니다.
- 위험 포인트:
  - “원시 HTML 삽입” 기능(예: React의 `dangerouslySetInnerHTML` 등)
  - 자동 이스케이프를 끄는 옵션
- 불가피하게 HTML을 렌더링해야 하면 **검증된 Sanitizer**(예: DOMPurify 같은 검증된 라이브러리)를 사용하고, 허용 태그/속성을 최소화합니다.

---

## 4) 입력 검증(Validation)과 정화(Sanitization)
출력 인코딩을 대체하진 못하지만, 공격면을 줄이는 데 유용합니다.

- **허용 목록(allowlist)** 기반으로 길이/문자/형식 제한(예: 닉네임은 `[a-z0-9_-]{1,20}`)
- “HTML을 허용하는 게시글” 같은 요구가 있다면:
  - 허용 태그/속성/프로토콜을 제한(예: `a[href]`에서 `javascript:` 차단)
  - 정화 결과를 **서버에서 저장**(클라이언트에서만 정화하면 우회될 수 있음)

---

## 5) CSP(Content Security Policy) 적용 (피해 완화 + 일부 차단)
CSP는 XSS가 발생했을 때 **악성 스크립트 실행을 어렵게** 만듭니다.

- 핵심 권장 사항:
  - `script-src`를 신뢰 도메인으로 제한
  - 인라인 스크립트 금지(가능하면 `'unsafe-inline'` 제거)
  - 가능하다면 **nonce/해시 기반**으로 필요한 스크립트만 허용
- 도입 팁:
  - 먼저 `Content-Security-Policy-Report-Only`로 영향 확인 → 점진적으로 차단 정책으로 전환

---

## 6) 쿠키/세션 보호 설정
XSS 자체를 “없애진” 못하지만, **탈취/오용 난이도를 크게 올립니다**.

- 세션 쿠키에 `HttpOnly` (JS에서 쿠키 접근 차단)
- `Secure` (HTTPS에서만 전송)
- `SameSite=Lax` 또는 상황에 따라 `Strict/None`(CSRF와 함께 고려)
- 가능하면 토큰을 `localStorage`에 저장하는 패턴 지양(탈취 위험 증가)

---

## 7) 보안 헤더 및 기타 설정
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy` 적절히 설정
- (레거시) `X-XSS-Protection`은 현대 브라우저에서 의미가 제한적이라 CSP/코드 수정이 더 중요

---

## 8) 개발 프로세스에서의 예방 (테스트/리뷰/모니터링)
- 코드 리뷰 체크포인트:
  - 사용자 입력이 HTML/속성/JS 컨텍스트로 들어가는 곳
  - “원시 HTML 렌더링” 사용 여부
- 보안 테스트:
  - SAST/DAST 도구, 보안 린터(예: ESLint 보안 규칙) 활용
  - OWASP ASVS/Top 10 기준으로 점검
- 로깅/탐지:
  - CSP report 수집, 이상 입력 패턴 모니터링

## 빠른 요약
1. **출력은 컨텍스트에 맞게 이스케이프**  
2. DOM 조작은 **`textContent` 중심**, `innerHTML` 최소화

## 개요

XSS 방지는 **다층 방어(Defense in Depth)** 전략이 필요합니다. 단일 방어책만으로는 충분하지 않으며, 여러 계층의 보안을 적용해야 합니다.

```
┌─────────────────────────────────────────────────────┐
│                    다층 방어 전략                     │
├─────────────────────────────────────────────────────┤
│  [입력 검증] → [출력 인코딩] → [CSP] → [쿠키 보안]    │
└─────────────────────────────────────────────────────┘
```

## 1. 입력 검증 (Input Validation)

### 1.1 화이트리스트 방식

> ⚠️ **핵심 원칙**: 허용된 것만 통과시키고, 나머지는 모두 차단

```javascript
// ❌ 블랙리스트 방식 (권장하지 않음)
function badValidation(input) {
    return input.replace(/<script>/gi, '');
    // 우회 가능: <scr<script>ipt>
}

// ✅ 화이트리스트 방식 (권장)
function goodValidation(input) {
    // 허용된 문자만 통과
    const allowedPattern = /^[a-zA-Z0-9가-힣\s]+$/;
    return allowedPattern.test(input);
}
```

### 1.2 입력 유형별 검증

| 입력 유형 | 검증 방법 | 정규식 예시 |
|----------|----------|------------|
| 이메일 | RFC 5322 준수 | `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` |
| 전화번호 | 숫자와 하이픈만 | `/^[\d\-]+$/` |
| 사용자명 | 영문, 숫자, 언더스코어 | `/^[a-zA-Z0-9_]{3,20}$/` |
| URL | URL 형식 검증 | `new URL(input)` 사용 |

## 2. 출력 인코딩 (Output Encoding)

### 2.1 컨텍스트별 인코딩

XSS 방지의 **가장 중요한 방어책**입니다. 데이터가 출력되는 위치에 따라 다른 인코딩이 필요합니다.

```
┌──────────────────┬─────────────────────────────────────┐
│    컨텍스트       │           인코딩 방법                │
├──────────────────┼─────────────────────────────────────┤
│ HTML 본문        │ HTML Entity 인코딩                  │
│ HTML 속성        │ HTML Attribute 인코딩               │
│ JavaScript      │ JavaScript 인코딩                   │
│ URL             │ URL 인코딩 (percent-encoding)       │
│ CSS             │ CSS 인코딩                          │
└──────────────────┴─────────────────────────────────────┘
```

### 2.2 HTML Entity 인코딩

```javascript
// 필수 변환 문자
const htmlEntities = {
    '&': '&amp;',
    '<': '&lt;',
    '>': '&gt;',
    '"': '&quot;',
    "'": '&#x27;',
    '/': '&#x2F;'
};

function encodeHTML(str) {
    return str.replace(/[&<>"'/]/g, char => htmlEntities[char]);
}

// 사용 예시
const userInput = '<script>alert("XSS")</script>';
const safeOutput = encodeHTML(userInput);
// 결과: &lt;script&gt;alert(&quot;XSS&quot;)&lt;&#x2F;script&gt;
```

### 2.3 컨텍스트별 인코딩 예시

```javascript
// 1. HTML 본문에 출력
document.getElementById('content').textContent = userInput; // ✅ 안전
document.getElementById('content').innerHTML = userInput;   // ❌ 위험

// 2. HTML 속성에 출력
element.setAttribute('data-value', encodeHTML(userInput));  // ✅

// 3. JavaScript 문자열에 출력
const safeJson = JSON.stringify(userInput);  // ✅

// 4. URL에 출력
const safeUrl = encodeURIComponent(userInput);  // ✅
```

### 2.4 템플릿 엔진의 자동 이스케이프

```html
<!-- Jinja2 (Python) -->
{{ user_input }}                    <!-- ✅ 자동 이스케이프 -->
{{ user_input | safe }}             <!-- ❌ 이스케이프 비활성화 - 위험! -->

<!-- EJS (Node.js) -->
<%= userInput %>                    <!-- ✅ 자동 이스케이프 -->
<%- userInput %>                    <!-- ❌ Raw 출력 - 위험! -->

<!-- Thymeleaf (Java) -->
<span th:text="${userInput}"></span>     <!-- ✅ 안전 -->
<span th:utext="${userInput}"></span>    <!-- ❌ 위험! -->
```

## 3. CSP (Content Security Policy)

### 3.1 CSP란?

CSP는 브라우저에게 **어떤 리소스를 로드하고 실행할 수 있는지** 알려주는 보안 정책입니다.

### 3.2 기본 CSP 설정

```http
# HTTP 응답 헤더
Content-Security-Policy: 
    default-src 'self';
    script-src 'self' https://trusted-cdn.com;
    style-src 'self' 'unsafe-inline';
    img-src 'self' data: https:;
    font-src 'self';
    connect-src 'self' https://api.example.com;
    frame-ancestors 'none';
    base-uri 'self';
    form-action 'self';
```

### 3.3 CSP 지시어 설명

| 지시어 | 설명 | 권장 값 |
|-------|------|--------|
| `default-src` | 기본 정책 | `'self'` |
| `script-src` | JavaScript 소스 | `'self'` (인라인 금지) |
| `style-src` | CSS 소스 | `'self'` |
| `img-src` | 이미지 소스 | `'self' data: https:` |
| `connect-src` | AJAX/WebSocket | `'self'` |
| `frame-ancestors` | iframe 허용 범위 | `'none'` |
| `base-uri` | base 태그 제한 | `'self'` |
| `form-action` | form 제출 대상 | `'self'` |

### 3.4 Nonce 기반 CSP (권장)

```http
Content-Security-Policy: script-src 'nonce-ABC123RandomValue'
```

```html
<!-- 서버에서 생성한 nonce 값 사용 -->
<script nonce="ABC123RandomValue">
    // 이 스크립트만 실행됨
    console.log('안전한 스크립트');
</script>

<!-- nonce가 없는 스크립트는 차단됨 -->
<script>
    alert('이 스크립트는 실행되지 않음');
</script>
```

### 3.5 Node.js/Express에서 CSP 적용

```javascript
const helmet = require('helmet');
const crypto = require('crypto');

app.use((req, res, next) => {
    // 요청마다 새로운 nonce 생성
    res.locals.nonce = crypto.randomBytes(16).toString('base64');
    next();
});

app.use(helmet.contentSecurityPolicy({
    directives: {
        defaultSrc: ["'self'"],
        scriptSrc: ["'self'", (req, res) => `'nonce-${res.locals.nonce}'`],
        styleSrc: ["'self'", "'unsafe-inline'"],
        imgSrc: ["'self'", "data:", "https:"],
        connectSrc: ["'self'"],
        fontSrc: ["'self'"],
        objectSrc: ["'none'"],
        frameAncestors: ["'none'"],
        baseUri: ["'self'"],
        formAction: ["'self'"]
    }
}));
```

}));

## 5. 쿠키 보안

### 5.1 보안 쿠키 속성

```http
Set-Cookie: sessionId=abc123; 
    HttpOnly;        # JavaScript 접근 차단
    Secure;          # HTTPS에서만 전송
    SameSite=Strict; # CSRF 방지
    Path=/;
    Max-Age=3600
```

### 5.2 속성별 설명

| 속성 | 역할 | XSS 방어 효과 |
|-----|------|-------------|
| `HttpOnly` | JS에서 `document.cookie` 접근 차단 | ⭐⭐⭐ **필수** |
| `Secure` | HTTPS에서만 쿠키 전송 | ⭐⭐ |
| `SameSite=Strict` | 같은 사이트에서만 쿠키 전송 | ⭐⭐⭐ |
| `SameSite=Lax` | GET 요청은 허용 | ⭐⭐ |

### 5.3 언어별 구현

```javascript
